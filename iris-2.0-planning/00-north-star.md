# 00 - 产品北极星

## 定位

Iris 2.0 的候选方向不是“给编辑器加高级表格”，而是在 Iris 现有本地 Markdown + AI Agent + 搜索/证据体系之上增加一个结构化知识对象层。

一句话：

> Markdown 写内容，Object 管属性，Collection 组织对象，View 提供观察方式，Agent 生成可确认的结构化变更。

## 为什么值得做

Iris 当前已经有多种半结构化实体：笔记、标签、双链、AI 收件箱、网页证据、附件、语料库、Agent 任务和版本系统。它们能被搜索和引用，但缺少一个用户可见、可组织、可计算、可被 Agent 安全操作的统一知识对象层。

Object / Collection / View 的价值在于：

- 将笔记、任务、资料、项目、人物等用户可见知识实体统一组织。
- 让 Agent 不只是读笔记和聊天，而能整理任务、资料和项目结构。
- 为多维表、统计、公式和高性能视图提供稳定底座。
- 保留 Markdown 文件主权，不把笔记改造成专有数据库文件。

## 非目标

Iris 2.0 不应成为以下产品：

- WPS / Excel 竞品。
- 企业协同多维表。
- Notion 克隆。
- Anytype 式完全自由对象宇宙。
- BI / 报表平台。
- 自动化工作流平台。
- 第三方插件市场或任意 MCP 工具平台。

## 产品原则

1. **写作优先**：用户仍然可以只写 Markdown，不必先建库。
2. **结构渐进浮现**：当内容需要组织时，Object、Collection、View 才出现。
3. **本地优先**：结构化数据保存在本地 SQLite，笔记正文保存在 `.md`。
4. **Agent 可控**：Agent 只生成 proposal，写入必须确认。
5. **性能优先**：多维表、公式和统计必须从第一天按大数据量设计。
6. **边界清楚**：多维表是 Grid View 能力，不是另一个数据库系统。

## 与 Anytype / AFFiNE 的关系

Anytype 证明“对象化笔记”方向成立；AFFiNE 证明“文档、白板、数据库、AI 融合工作区”方向成立。Iris 可以借鉴二者，但不应复制二者：

- 借鉴 Anytype 的对象 / 关系 / 集合思路。
- 借鉴 AFFiNE 的多形态工作区思路。
- 保留 Iris 的 Markdown 主权和 Agent 安全写入边界。

## 成功标准

- 用户无需理解对象模型，也能继续写普通 Markdown。
- 用户需要结构化时，可以自然创建任务库、资料库、项目库和主题集合。
- Agent 能从笔记和资料中生成结构化 proposal。
- 用户能审查、确认、回滚重要写入。
- 大型 Grid View 明显快于典型云端 Notion database 体验。
