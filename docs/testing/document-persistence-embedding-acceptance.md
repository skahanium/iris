# v1.2.9 文档持久化与嵌入验收

本文件定义 v1.2.9 的交付门槛，不是“已通过”记录。任何 PR、发布说明或路线图只有附上对应命令输出和 Windows 真机证据后，才能把下列项目表述为已完成。

## 不可变的成功语义

- `.md` 是笔记内容的唯一权威来源。只有读取成功或当前 revision 的 Markdown 原子写入成功，才能建立已保存基线；路径变化、编辑器重挂载、Tab 缓存或 SQLite 索引都不是保存回执。
- `FileWriteResult.indexStatus = degraded` 仍是 Markdown 保存成功。索引和嵌入均为可重建派生数据，失败时不能回滚、删除或遮蔽已落盘正文。
- 关闭应用、关闭标签、切换库和安装更新都必须等待同一个持久化屏障；任一 dirty revision 无法确认落盘时，操作失败且窗口、标签和内存编辑保持可用。
- 嵌入失败不影响 Markdown 编辑、保存或关键词检索。自动升级只在初始索引完成、没有 dirty 文档且连续空闲 30 秒后对 `legacy_ready` generation 尝试一次；失败后跨启动不自动重试。启动扫描不得把当前模型、维度、来源指纹和向量长度均完整覆盖的 `ready` generation 降级，也不得为它加载模型。

## 自动化验收

必须新增或保留可重复运行的测试，覆盖以下闭环：

1. 文档协调器：标题、正文、AI 修改和版本恢复都先捕获完整 Markdown revision；自动保存与手动保存重叠、旧回执晚到、多个 dirty 标签和同一路径合并写入时，旧回执不能清除较新的 dirty 状态。
2. 重命名：重命名前后持续输入、标题同步重命名和编辑器重挂载期间的保存，都只在最新 revision 落盘后移动路径。新路径不能把内存快照直接当作基线。
3. 生命周期：可信 remount 快照可在编辑器未就绪时写入；没有可信快照时 `barrier`、关闭、标签关闭、切换库和更新安装都失败并保留编辑。写入失败后重试必须可恢复。
4. 写入服务：原子替换使用同目录唯一临时文件并同步；失败保留旧文件、并发临时文件互不冲突。普通保存、创建、AI 应用、版本恢复、模板、分类迁移/导出和链接级联都走统一服务。索引异常返回 `degraded`，不回滚正文，并排队修复。
5. `054` 迁移：up/down 均可验证；零进度旧 `rebuilding` 变为 `legacy_ready`，有进度的中断任务变为带安全失败码的 `failed`。启动残留 `running`/`paused`/旧 `rebuilding` 必须先重新统计当前模型、维度、来源指纹和向量长度的覆盖：完整 generation 恢复为 `ready` 并清除错误，缺失或失配才变为 `interrupted_restart`，且不触发无限自动重试。
6. 嵌入调度器：重复启动返回 `already_running`；一次自动空闲启动、手动暂停/恢复、模型缺失、安全失败码、进度事件、部分向量续建和内容未变但覆盖缺失的增量修复均有测试。完整 generation 经启动扫描或索引提交通知后必须保持 `ready` 且不加载模型；相同模型、维度和来源指纹的有效向量必须保留，未知或不匹配项必须重算。
7. 连接边界：使用阻塞式 `EmbeddingBatcher` 测试，确认模型推理进行时普通 `file_write` 仍能及时完成，证明推理没有持有 SQLite 连接。

## Windows 真机 E2E（发布前必须完成）

无头单元测试不能代替此项。使用 Windows 临时 vault 和真实 Tauri 应用，记录命令、构建类型、系统版本与结果，但不得记录标题、正文、路径或模型原始错误。

1. 新建空白文档，输入唯一标题与多段正文。
2. 修改标题触发文件名同步重命名；在编辑器重挂载和保存防抖窗口内继续输入，并立即保存、关闭应用。
3. 重启应用，直接从磁盘读取重命名后的 `.md`，再由应用打开。标题和正文必须逐字一致。
4. 覆盖多标签 dirty、重命名前后继续输入、写入失败后重试、无可信快照关闭失败、索引失败但 Markdown 成功，以及安装更新的持久化屏障。

在此闭环真实执行并留存证据前，状态只能写作“待验收”，不能声称 Windows E2E 已通过。

自动化入口为 `npm run test:desktop:windows`。它必须在真实 Tauri WebView 中确认重命名后的新 `[data-path]` editor surface 已稳定进入 `visible`，明确将选择定位到文末，输入唯一正文并立即保存、关闭、重启，再先作磁盘字节断言并通过最近笔记 UI 重新打开断言标题与全文。E2E 不捕捉 React 生命周期的瞬时 `staging` 帧；重挂载期间保存的快照安全约束由确定性的组件与持久化生命周期测试覆盖。该入口会在 PR CI 与发布打包 workflow 中运行；PR 是否因该检查而禁止合并，取决于仓库外 GitHub 分支保护规则是否将该状态设为 required。发布 workflow 内的 Windows 打包步骤则在该 E2E 成功前不会上传工件。静态 Vitest 契约不能替代其 Windows 运行日志。

### Windows E2E 的稳定边界

真实 Windows E2E 必须等待重命名后的目标 editor surface 以 `visible` 状态稳定挂载后再触发保存，并验证保存、关闭、第二次启动、磁盘 Markdown 字节和 UI 重新打开的完整链路。它不得轮询或要求捕获 React 生命周期中的瞬时 `staging` 帧：WebDriver 的轮询与平台调度不能可靠观察该帧。`staging` 期间不得以空白状态覆盖已加载文档、且可恢复快照必须用于保存的约束，由确定性的组件与持久化生命周期测试覆盖。E2E 失败日志仅输出白名单稳定错误码，不记录正文、标题、路径或原始 WebDriver 错误。

## 隐私与可观测性

日志、指标、事件、错误对话和验收记录只允许修订号、耗时区间、稳定结果码、重试次数与嵌入数量。禁止记录或展示 Markdown 正文、标题、路径、模型原始错误、API Key、token 或解密后的涉密内容。

## 发布门禁

在上述场景有通过证据后，运行并检查完整输出：

```bash
cargo fmt --all -- --check
cargo clippy --all-targets -- -D warnings
cargo test
npm run lint
npm run format:check
npm run typecheck
npm run test
npm run audit:rust
npm audit
npm run test:e2e
```

任一命令失败、缺少 Windows 真机闭环，或只完成分散单元测试时，都不能宣称 v1.2.9 的本链路已完成。
