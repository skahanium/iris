## 1. 背景与动机

### 1.1 Iris RAG 定位

Iris 是桌面端本地优先的 Markdown 笔记应用，AI 系统的核心价值在于**从用户的笔记库中精准检索相关内容，辅助写作与知识回顾**。RAG（Retrieval-Augmented Generation）管线直接决定 AI 输出的事实准确性和引用可靠性——这不是"锦上添花"功能，而是 AI 系统的质量基石。

### 1.2 为何现在优化

当前 RAG 管线（v1.2.3 基线）已具备完整的工程骨架：5 层混合检索、语义锚点、法规条款索引、可溯源证据包。但在深度使用场景下，暴露出若干结构性问题：

- **笔记元数据未融入检索**：tag、frontmatter 自定义字段、时间信息被索引但对检索毫无影响
- **图谱检索层空跑**：`block_links` 无自动填充机制，图谱层在实际使用中大概率返回空
- **Embedding 模型与中文场景不匹配**：AllMiniLML6V2 是英文优先的轻量模型
- **缺少精排**：所有排序依赖硬编码权重乘法，无交叉编码器精排
- **分块策略粗糙**：无 overlap、无语义感知分块

这些问题在笔记库规模增长、查询复杂度提高时将逐渐恶化。v1.2.6 的目标是系统性地解决这些差距，使 RAG 质量从"可用"提升到"可靠"。

## 2. 现状评估

### 2.1 架构概览

```
.md 文件
  → FileWatcher (500ms debounce) / index_vault_incremental (content_hash 跳过)
  → frontmatter.rs: parse_note() → title, tags, body
  → wikilink.rs: index_wiki_links() → links 表
  → fts.rs: upsert_fts() → files_fts (CJK bigram)
  → chunker.rs: chunk_markdown(body, 2000) → chunks 表 (heading_path, source_span, hash)
  → EmbedQueue → fastembed AllMiniLML6V2 (384d) → chunk_embeddings + vec_chunks

用户查询
  → context_planner: detect_intent() (关键词匹配)
  → packet_builder: build_context_packets()
    → retrieval_broker: 5 层混合检索
      Layer 1: FTS5 关键词 (files_fts)
      Layer 2: 向量搜索 (vec_chunks + vec_anchors + vec_regulations)
      Layer 3: 图谱邻居 (block_links WHERE is_confirmed = 1)
      Layer 4: 精确法规条款 (regulation_index)
      Layer 5: 文体模板 (genre_templates)
    → rank.rs: fuse_and_rank() (层权重 * corpus 角色权重)
    → filter_packets_by_scope()
  → prompt_builder: 注入系统提示词

LLM 生成
  → 可通过 search_keyword / search_semantic 工具追加检索
  → evidence_ledger: 去重 + 引用标签分配
```

### 2.2 做对了什么（优势）

| 维度                | 评价                                                                                               |
| ------------------- | -------------------------------------------------------------------------------------------------- |
| **混合检索架构**    | 5 层 broker 设计良好，层之间松耦合，易于独立替换或增加新层                                         |
| **语义锚点**        | 超越 chunk 粒度的结构化知识单元（Claim/Definition/Decision/Fact），领先于多数笔记应用 RAG          |
| **Corpus 角色加权** | authority/exemplar/reference/lookup 分层权重，对专业写作场景有意义                                 |
| **可溯源证据链**    | source_path + heading_path + source_span(字节偏移) + content_hash + citation_label，完整的引用链路 |
| **渐进降级**        | sqlite-vec 为可选 feature，不可用时回退 Rust cosine 扫描（上限 8000 chunks）                       |
| **CJK 优化**        | FTS5 unicode61 + CJK bigram 增强，对中文搜索有针对性处理                                           |
| **增量索引**        | content_hash 跳过未变文件 + 前台让步(8ms sleep) + cache 自动失效                                   |
| **评测框架**        | semantic_recall_eval.rs 有完整的 Recall@5/MRR@10 评估基础设施（20 条 fixture）                     |

### 2.3 成熟度评分

**总分：7/10** — 已超越"demo RAG"阶段，具备生产级管线骨架，但在检索精度、元数据利用、图谱落地三个维度存在系统性差距。

## 3. 差距分析

### 3.1 笔记体系与 RAG 的协同断裂

这是最核心的问题：Iris 作为笔记应用，笔记体系本身提供了丰富的结构化信号（tag、link、frontmatter、时间、层级），但 RAG 管线几乎完全忽略了这些信号。

#### 3.1.1 Tag 未参与检索 [严重程度：中高]

**现状**：`tags` + `file_tags` 表有数据（从 frontmatter YAML 和正文 `#tag` 提取），`sync_file_tags()` 正常工作。但 5 个检索层（FTS / 向量 / 图谱 / 精确 / 模板）均不查询 tag 表。

**影响**：

- 用户无法按 tag 缩小 AI 搜索范围（如"只搜 #纪律处分 相关的笔记"）
- Tag 对排序无贡献——同样相关的两篇笔记，一篇有精确 tag 匹配，一篇没有，排名相同
- `RetrievalScope` 支持路径前缀和精确路径，但不支持 tag 过滤

**根因**：`RetrievalScope` 设计时只考虑了路径维度，tag 维度从未纳入。

#### 3.1.2 links 与 block_links 双轨脱节 [严重程度：中]

**现状**：

- `links` 表：由 `index_wiki_links()` 自动填充，存储 `[[wikilink]]` 关系，用于 backlink 展示和系统提示词注入
- `block_links` 表：图谱检索层查询的目标表（`WHERE is_confirmed = 1`），但**无自动填充机制**
- `confirm_block_link` 工具注册在 catalog 但缺乏完整执行路径

**影响**：图谱检索层（Layer 3）在实际使用中大概率返回空结果，相当于空跑。wikilink 关系只服务于 backlink 展示，未融入检索排序。

**根因**：`block_links` 被设计为 AI 建议 + 用户确认的精细链接，但填充通路未完成；同时 `links` 表的数据完全未用于检索。

#### 3.1.3 无时间衰减信号 [严重程度：中]

**现状**：`files.updated_at` 和 `files.created_at` 已索引但从未参与排序。`rank.rs` 的 `fuse_and_rank()` 仅考虑层权重和 corpus 角色权重。

**影响**：对于活跃笔记库，最近编辑的笔记往往更相关，但与两年前的旧笔记在相同相关性下地位等同。

#### 3.1.4 Frontmatter 自定义字段被忽略 [严重程度：低-中]

**现状**：完整 YAML frontmatter 以 JSON 存储于 `files.frontmatter`，但除 `title` 和 `tags` 外，所有自定义字段（author、aliases、category、source）对 RAG 无影响。

**影响**：用户精心维护的元数据无法辅助检索。例如 `aliases: [纪律条例, 处分条例]` 不影响搜索匹配。

### 3.2 检索质量瓶颈

#### 3.2.1 Embedding 模型与中文场景不匹配 [严重程度：高]

**现状**：使用 `fastembed` 的 `AllMiniLML6V2`（384 维），一个**主要为英文训练的轻量模型**。

**影响**：

- 对中文法规文本的语义区分度有限（"组织纪律" vs "政治纪律" 可能得到相近的向量表示）
- 384 维度在区分细粒度语义差异时容量不足
- 模型**硬编码**在 `engine.rs` 中（`EmbeddingModel::AllMiniLML6V2`），无配置化切换机制

**量化数据**：当前 fixture vault 评测集 Recall@5 = 1.00，但 fixture 设计为"主题互不混淆"的理想场景。在实际笔记库中，同领域笔记间的语义区分是真正的挑战，AllMiniLML6V2 在这类场景下的 Recall 预计会显著下降。

#### 3.2.2 无 Reranker [严重程度：中-高]

**现状**：`CapabilitySlot` 枚举中有 `"reranker"` 占位，但无实现。排序完全依赖硬编码的层权重乘法。

**影响**：

- 初检结果中"相似但不相关"的噪声无法被精排过滤
- 对于同一法规的不同条款，粗排可能将错误条款排在前面
- 跨层的分数不可比（FTS 的 0.7 和向量的 0.8 含义不同），简单乘法无法真正统一

**业界对比**：现代 RAG 系统几乎都采用 two-stage（粗检 + 精排）架构。精排（cross-encoder reranker）通常可将 Recall@5 提升 10–20 个百分点。

#### 3.2.3 分块策略粗糙 [严重程度：中]

**现状**（`indexer/chunker.rs`）：

- 按 heading 行 + 空行分割
- 最大 2000 chars，最小 100 chars
- **无重叠（overlap）**：相邻 chunk 之间没有共享文本
- **无语义感知**：纯结构分割，不考虑段落语义完整性

**影响**：

- 信息可能在 chunk 边界处被截断，导致关键上下文丢失
- 长段落（无 heading 的连续文本）只在空行处切割，可能产生语义不完整的 chunk
- 没有 parent-child chunk 机制：检索到的小 chunk 缺少上下文补充

#### 3.2.4 Query 理解层过于 Heuristic [严重程度：中]

**现状**（`context_planner.rs`）：

- 意图检测完全基于关键词匹配（`contains("条例")`、`contains("比较")`）
- 子查询生成为硬编码模板（`format!("{} 相关规定", query)`）
- 无 LLM-based query expansion

**影响**：

- 复杂/隐晦查询分类能力有限（如"这段话有没有问题"不会被识别为一致性检查）
- 无法从用户意图推断出更好的检索关键词
- 子查询质量受限于预设模板

### 3.3 工程质量差距

#### 3.3.1 Embedding 模块零测试 [严重程度：高]

| 函数                                | 性质                           | 测试 |
| ----------------------------------- | ------------------------------ | ---- |
| `cosine_similarity()`               | 纯函数                         | 无   |
| `f32_to_bytes()` / `bytes_to_f32()` | 量化编解码，数据完整性关键路径 | 无   |
| `truncate_snippet()`                | 纯函数                         | 无   |
| `store_chunk_embeddings()`          | 嵌入写入路径                   | 无   |
| `EmbedQueue` batching/dedup         | 并发逻辑                       | 无   |

量化编解码的 round-trip 错误会导致所有向量搜索结果静默错误，且极难定位。

#### 3.3.2 检索层测试覆盖不均

| 检索层                                    | 单元测试 | 评估                                       |
| ----------------------------------------- | -------- | ------------------------------------------ |
| FTS (`retrieval_broker/fts.rs`)           | 0 个     | `escape_fts5_query` 是安全敏感函数，无测试 |
| Vector (`retrieval_broker/vector.rs`)     | 0 个     | 3 个搜索函数无测试                         |
| Graph (`retrieval_broker/graph.rs`)       | 0 个     | SQL 查询和 packet 构建无测试               |
| Template (`retrieval_broker/template.rs`) | 0 个     | SQL 查询无测试                             |
| Exact (`retrieval_broker/exact.rs`)       | 1 个     | 仅测 regex，不测 DB 查询                   |
| Rank (`rank.rs`)                          | 2 个     | 基本够用，缺去重边界测试                   |

**对比**：`indexer/scan.rs`（22 个测试）和 `session_evidence.rs`（6 个高质量行为测试）覆盖良好。检索层的测试投入与其重要性不成比例。

#### 3.3.3 评测套件未集成 CI

`semantic_recall_eval.rs` 有完整的 Recall@5 评估框架和 20 条 fixture，但所有实质性测试标记为 `#[ignore]`（需下载 embedding 模型）。这意味着：

- 检索质量无法持续监控
- 代码变更可能静默降低检索质量
- 无法在 PR 中自动检测回归

## 4. 优化方案总览

### 4.1 设计原则

1. **笔记优先**：所有优化以"更好地服务笔记体系"为目标，而非追求通用 RAG 能力
2. **增量推进**：每个 Phase 独立可发布，不做一次性大重构
3. **可测量**：每项改进必须可通过评测集量化验证
4. **本地优先**：优先使用本地模型和计算，LLM API 调用仅用于不可替代的环节
5. **兼容性**：schema 变更走增量 migration，旧数据自动兼容


## 5. Phase 1 — 低成本高收益（1–2 周）

### P1.1 Tag 融入检索

**目标**：用户的 tag 标注参与 RAG 检索排序和过滤。

**方案 A：Tag 后过滤（推荐先实现）**

在 `RetrievalScope` 中增加 tag 维度：

```rust
// retrieval_scope.rs
pub struct RetrievalScope {
    pub path_prefixes: Vec<String>,
    pub exact_paths: Vec<String>,
    pub required_tags: Vec<String>,    // 新增：结果必须包含这些 tag
    pub preferred_tags: Vec<String>,   // 新增：包含这些 tag 的结果获得加权
}
```

`filter_packets_by_scope()` 增加 tag 过滤逻辑，需要在 `ContextPacket` 上新增 `tags: Vec<String>` 字段，由各检索层在构建 packet 时 JOIN `file_tags` 填入。

**方案 B：Tag 加权（与 A 同步实现）**

在 `rank.rs` 的 `fuse_and_rank()` 中，对 tag 匹配的 packet 施加加权：

```rust
fn tag_boost(packet: &ContextPacket, scope: &RetrievalScope) -> f64 {
    if scope.preferred_tags.is_empty() { return 1.0; }
    let matches = packet.tags.iter()
        .filter(|t| scope.preferred_tags.contains(t))
        .count();
    1.0 + 0.05 * matches as f64  // 每个匹配 tag +5%
}
```

**前端集成**：`AiComposer` 的 `@` 语料库选择器已支持路径范围；增加 `#tag` 范围选择，复用 `ContextScopeChips` 组件。

**涉及文件**：

- `src-tauri/src/ai_runtime/retrieval_scope.rs`
- `src-tauri/src/ai_runtime/retrieval_broker/rank.rs`
- `src-tauri/src/ai_runtime/retrieval_broker/*.rs`（各层 packet 构建时 JOIN file_tags）
- `src-tauri/src/ai_runtime/packet_builder.rs`
- `src/components/ai/AiComposer.tsx`（`#tag` 输入）

**验收标准**：

- 带 `required_tags` 的检索请求只返回含指定 tag 的结果
- 带 `preferred_tags` 的检索请求，tag 匹配的结果排序提升
- 有对应的 Rust 单元测试

### P1.2 Links 降级为图谱检索数据源

**目标**：让图谱检索层（Layer 3）基于 wikilink `links` 表工作，不再依赖几乎为空的 `block_links` 表。

**方案**：

修改 `retrieval_broker/graph.rs` 的 `search_graph_neighbors()` 函数，当 `block_links` 为空时自动降级查询 `links` 表：

```rust
pub(super) fn search_graph_neighbors(
    conn: &Connection,
    file_id: i64,
    max_results: usize,
) -> Vec<ContextPacket> {
    // 优先查 block_links（精确的 block 级链接）
    let block_results = search_block_links(conn, file_id, max_results);
    if !block_results.is_empty() {
        return block_results;
    }
    // 降级查 links（wikilink 级链接）
    search_wikilink_neighbors(conn, file_id, max_results)
}
```

`search_wikilink_neighbors()` 查询双向链接（outbound + inbound），构建 ContextPacket，score 固定为 0.5（低于 block_links 的 confidence）。

**涉及文件**：

- `src-tauri/src/ai_runtime/retrieval_broker/graph.rs`

**验收标准**：

- 有 wikilink 但无 block_links 时，图谱层返回 wikilink 邻居
- 有 block_links 时，优先使用 block_links
- 新增 2-3 个 in-memory DB 单元测试

### P1.3 时间衰减加权

**目标**：最近更新的笔记在检索排序中获得轻微加权。

**方案**：

在 `rank.rs` 增加时间衰减因子：

```rust
fn recency_weight(packet: &ContextPacket) -> f64 {
    // 需要在 ContextPacket 中新增 updated_at 字段
    let Some(updated_at) = &packet.updated_at else { return 1.0 };
    let age_days = (now() - updated_at).num_days().max(0) as f64;
    // 半衰期 180 天：6 个月前的笔记权重降至 ~0.5 * boost
    // 加权范围 [1.0, 1.10]，不会大幅颠覆相关性排序
    1.0 + 0.10 * (-age_days / 180.0_f64).exp2()
}
```

设计考量：

- 加权范围控制在 1.0–1.10，确保时间信号**辅助**而非**主导**排序
- 半衰期 180 天适合笔记场景：比实时新闻长，比学术论文短
- 在 `fuse_and_rank()` 中与 `layer_weight * corpus_role_weight` 一起乘入

**涉及文件**：

- `src-tauri/src/ai_runtime/retrieval_broker/rank.rs`
- `src-tauri/src/ai_runtime/mod.rs`（ContextPacket 增加 `updated_at`）
- 各检索层的 packet 构建（JOIN `files.updated_at`）

**验收标准**：

- 内容相关性相同时，更新更近的笔记排名靠前
- 加权不超过 10%，不会颠覆内容相关性主导地位
- 新增单元测试验证衰减曲线

### P1.4 Embedding 纯函数测试

**目标**：补齐 `embedding/engine.rs` 中纯函数的单元测试。

**测试清单**：


### P2.2 Chunk Overlap

**目标**：相邻 chunk 之间增加文本重叠，避免边界处语义丢失。

**方案**：

### P2.3 Parent-Child Chunk 检索

**目标**：检索时返回小 chunk 以保精度，展示时返回父段落以保完整性。

**方案**：

在 `chunks` 表增加 `parent_chunk_id` 字段（migration），将 heading 级别的大段落作为 parent chunk 存储。检索命中子 chunk 后，通过 `parent_chunk_id` JOIN 获取完整段落，填入 `ContextPacket.excerpt` 的扩展区域。

或者，利用现有的 `source_start` / `source_end` 字节偏移，在构建 packet 时**按需从原文件扩展上下文窗口**（向前向后各扩展 500 chars，以段落边界截断），这种方案不需要 schema 变更：

```rust
fn expand_context(file_content: &str, start: usize, end: usize, window: usize) -> String {
    let expanded_start = file_content[..start]
        .rfind("\n\n").map(|p| p + 2).unwrap_or(0)
        .max(start.saturating_sub(window));
    let expanded_end = file_content[end..]
        .find("\n\n").map(|p| end + p).unwrap_or(file_content.len())
        .min(end + window);
    file_content[expanded_start..expanded_end].to_string()
}
```

**推荐**：先实现基于 source_span 的按需扩展（无 schema 变更），后续评估是否需要 parent_chunk_id。

**涉及文件**：

- `src-tauri/src/ai_runtime/packet_builder.rs`
- `src-tauri/src/ai_runtime/model_gateway_impl.rs`（evidence 格式化时扩展）

**验收标准**：

- 检索结果的 excerpt 包含完整的段落上下文
- 不影响检索精度（检索仍基于原始 chunk）

### P3.1 Reranker 集成

**目标**：在粗检结果之上增加交叉编码器精排，显著提升排序精度。

**技术路线**：

**方案 A：本地 Cross-Encoder（推荐）**

使用 `ort`（ONNX Runtime）加载轻量级 cross-encoder 模型：

```rust
pub trait Reranker: Send + Sync {
    fn rerank(&self, query: &str, documents: &[&str]) -> Vec<f64>;
}

pub struct OnnxReranker {
    session: ort::Session,
    tokenizer: tokenizers::Tokenizer,
}
```

候选模型：

- `cross-encoder/ms-marco-MiniLM-L-6-v2`（~80MB，英文）
- `BAAI/bge-reranker-v2-m3`（~570MB，多语言含中文）
- `BAAI/bge-reranker-base`（~260MB，中文优先）

集成点：在 `rank.rs` 的 `fuse_and_rank()` 之后、`diversify_by_source_path()` 之前插入 reranker：

```rust
pub fn fuse_rank_and_rerank(
    packets: &mut Vec<ContextPacket>,
    query: &str,
    max_results: usize,
    reranker: Option<&dyn Reranker>,
) {
    fuse_and_rank(packets, max_results * 3); // 粗排取 3x
    if let Some(reranker) = reranker {
        let docs: Vec<&str> = packets.iter().map(|p| p.excerpt.as_str()).collect();
        let scores = reranker.rerank(query, &docs);
        for (p, s) in packets.iter_mut().zip(scores) {
            p.score = s as f32; // 用 reranker 分数替换
        }
        packets.sort_by(|a, b| b.score.partial_cmp(&a.score).unwrap_or(Ordering::Equal));
    }
    diversify_by_source_path(packets, max_results);
}
```

**方案 B：LLM-based Rerank（作为补充）**

对 Top-10 结果调用 LLM 进行 listwise rerank。成本较高，适合 ResearchSynthesis 场景。

**涉及文件**：

- `src-tauri/src/ai_runtime/retrieval_broker/rank.rs`

## 10. 风险与约束

### 10.1 技术风险

| 风险                                 | 影响                  | 缓解措施                                       |
| ------------------------------------ | --------------------- | ---------------------------------------------- |
| 新 embedding 模型增加应用体积        | BGE-small-zh 约 +70MB | 首次启动时下载，非打包内置；或作为可选 feature |
| Reranker ONNX Runtime 增加编译复杂度 | 新增 C++ 依赖         | 先评估 `ort` crate 的跨平台构建状况            |
| Schema 变更导致旧数据需迁移          | 用户升级体验          | 所有变更走增量 migration + 后台异步重建        |
| Embedding 模型切换导致向量空间不兼容 | 必须全量重建          | 后台队列处理，不阻塞用户操作                   |
| Chunk overlap 增加存储和索引时间     | 约增加 5–10%          | 可接受范围                                     |

### 10.2 约束条件

- **AGPL-3.0 兼容**：所有新增依赖（`ort`、reranker 模型）必须许可证兼容
- **本地优先**：embedding 和 reranker 必须完全本地运行，不依赖外部 API
- **sqlite-vec 可选**：所有优化不依赖 sqlite-vec 为 default feature
- **增量 migration**：Schema 变更必须有 up/down 回滚脚本
- **向后兼容**：旧 vault 升级后自动触发后台重建，不要求用户手动操作
- **跨平台**：方案必须在 macOS / Windows / Linux 三平台可行

### 10.3 决策点

以下决策需要在实施前确认：

1. **默认 embedding 模型**：继续 AllMiniLML6V2 还是切换 BGE-small-zh？需要在扩展评测集上对比。
2. **Reranker 部署方式**：打包 ONNX 模型还是首次使用时下载？需要评估 `ort` 在三平台的构建状况。
3. **Chunk overlap 参数**：50 / 100 / 200 chars？需要 A/B 评测确定最优值。
4. **图谱层长期策略**：`block_links` 的 AI 自动填充机制 vs 直接复用 `links` 表？涉及产品决策。
