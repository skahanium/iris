# v1.2.6 RAG 检索优化设计

**状态：已批准，实施中**  
**版本范围：v1.2.6（均衡版）**  
**排期权威：** [ROADMAP.md](../../ROADMAP.md)

## 1. 目标与原则

本版本提高 Iris 本地 RAG 的正确性、中文检索质量、可观测性和可复现实测能力。它首先修复现有 broker 的候选/作用域/证据契约，再统一嵌入模型和索引代际，最后以 Rank v2 改善排序。所有派生数据可由 `.md` 重建；不改写用户笔记。

本版本交付：

- BGE-small-zh-v1.5（512 维）统一强制迁移；模型作为约 95MB 安装包资源内置，不提交二进制到 Git。
- 可恢复的索引代际切换、进度状态和失败降级。
- 正确的 broker 作用域、tags/aliases 元数据检索、真实文本证据和 Rank v2。
- 端到端 fixture、固定基线、质量/性能/体积门槛与 CI 分层验证。

## 2. 已知基线问题

1. broker 在 fusion 后才做 scope filter，范围外候选可耗尽 Top-K。
2. 普通 `semantic_search` 与 broker 的 sqlite-vec 不可用语义不一致；后者只标记未就绪，未明确使用同等降级策略。
3. 真实调用在 `corpus_role` 标注前排序，使角色权重失效。
4. 当前“normalize”实为固定原始分数相乘；不同候选源分数不可直接比较。
5. sqlite-vec 的 `float[384]` 不能安全复用现有量化 BLOB；写入失败被忽略会产生“索引已建”的假象。
6. 链接候选会形成“linked via”说明而非真实来源文本；tags/aliases 未进入完整 scope 与 metadata 检索链路。
7. 当前 ignored semantic eval 只测 embedding，不覆盖真实 broker、scope 或 ContextPacket。

## 3. 嵌入与索引迁移

### 3.1 模型交付

使用 fastembed 支持的 `BGESmallZHV15`，模型文件由受控 `model:prepare` 脚本按锁定 URL、revision、SHA-256 和许可证清单下载到被 `.gitignore` 忽略的 staging 目录；Tauri bundle resources 从该目录打包。CI 缓存下载结果。模型二进制不得进入 Git。

中文检索 query 使用模型要求的检索指令前缀；文档 representation 为：标题、heading path、aliases/tags 和规范正文。最终 `ContextPacket` 仍引用原始正文与稳定 span/hash。

### 3.2 代际与数据库

新增：

- `044_embedding_generation_v2.sql` 及对应 down：嵌入 generation、模型 id、维度、内容 hash、状态/错误元数据；分离 legacy 读取与 v2 原始 float32 向量写入。
- `045_metadata_fts.sql` 及对应 down：`files_metadata_fts(path, aliases, tags)`，不污染正文 FTS 的 snippet 语义。

BGE v2 重建覆盖所有派生嵌入：`chunk_embeddings`/`vec_chunks`、`semantic_anchors`/`vec_anchors`、`regulation_index`/`vec_regulations` 和 `skill_activation_index.embedding_json`。sqlite-vec 写入使用符合维度的原始 float32 BLOB，任何写入错误必须向上返回并令 generation 失败。

迁移期间旧代际保持只读兼容：若旧模型缓存不可用，仅使用 FTS/metadata/graph/runtime 非向量候选，不重新下载旧模型。新建/修改笔记标记为 v2 pending；仅当覆盖数、内容 hash 和 generation 均完整时原子切换 active generation，绝不混合不同维度向量。

### 3.3 状态、重试与 UI

定义 `legacy_ready | rebuilding | ready | failed` 状态，提供 `search_embedding_status()` 和 `embedding-index-progress` 事件。DTO 至少包含 active/target model、dimension、indexed/total、phase、已脱敏 lastError。`search_reindex()` 保持无参 IPC 签名：pending/failed generation 时重试模型构建，否则按现有增量重建逻辑执行。管理中心和搜索入口显示状态、进度与可重试错误；重建不得造成主线程超过 50ms 的连续阻塞。

## 4. 检索正确性与元数据

1. 在 fusion 之前应用 `paths`、`pathPrefixes`、corpus 和新增 `requiredTags`（AND 语义）scope，空候选时才返回空。
2. 在 rank 前完成 corpus role 注释；所有向量、FTS、metadata、graph、runtime 候选使用统一候选契约和诊断。
3. frontmatter 解析 `aliases`（单字符串或字符串列表）和 tags；runtime documents 同样解析 frontmatter/正文 tags。
4. 图谱只扩展邻接文件，再从这些文件检索真实 chunks；`ContextPacket` 不得将“存在链接”当作证据正文。
5. chunk 命中根据段落边界扩展前后最多 400 字符，UTF-8 安全并受 prompt 预算约束。
6. exact regulation 候选先按 scope 过滤并固定置顶；所有非精确业务加权均设上限。

## 5. Rank v2

### 5.1 评分

候选源先各自按排名转换为加权 RRF，`k = 60`：

| 来源                        | 权重 |
| --------------------------- | ---- |
| runtime / vector-regulation | 1.20 |
| FTS / vector-chunk          | 1.00 |
| anchor                      | 0.90 |
| metadata                    | 0.80 |
| graph / template            | 0.70 |

结构化加分总上限为 1.15，所有非精确业务加权总上限为 1.25。然后用 MMR（`λ = 0.75`）去重；Top-10 每一来源文件最多 2 个包，候选不足时再回填。

### 5.2 reranker 边界

引入可测试的 `CandidateReranker` 接口、Noop 实现和 fake 测试实现，为日后可选 reranker 留接口。本版本不下载或打包交叉编码器，不调用 LLM rerank，也不引入 1GB 级模型负担。

## 6. 评测、性能与交付

建立 `docs/eval/fixtures/rag-v2-vault/` 的 48 篇合成笔记与 60 条标签查询；运行端到端 `hybrid_retrieve → Rank → scope → ContextPacket`。固定基线文件为 `docs/eval/results/v1.2.5-hybrid.json`。

发布必须满足：scope 泄漏 0、span/hash 有效性 100%、候选 Recall@30 ≥ 0.95、Recall@5 ≥ 0.80、无答案 FPR ≤ 0.10；相对基线 nDCG@10 和 MRR@10 各提升 ≥ 0.05，任一标签子集不回退超过 0.02。相同机器 warm p95 不差于基线 25%（目标 ≤500ms）；Windows/macOS 包体积 A/B 记录且增量 ≤110MB。

CI 分为前端检查、Rust fmt/clippy/test、默认 RAG eval、Linux sqlite-vec parity 与文档检查。最终通过完整质量检查、Windows/macOS 打包与 `npm run version:check` 后才可发布。

## 7. 明确非目标

v1.2.6 不包含语义分块、chunk overlap、HyDE、LLM query expansion、全局时间衰减、LLM rerank、交叉编码器模型、第三方向量数据库或对用户 `.md` 的格式迁移。这些议题必须在后续 ROADMAP 里单独排期和评测。
