# 模型与供应商注册中心设计

## 背景

当前“模型与供应商”页面把供应商凭据、模型 ID、能力路由和诊断按钮放在同一张供应商卡片里。后端诊断优先请求供应商 `/models`，只要该端点成功，就可能把单个模型的“诊断”判定成功；因此即使用户输入错误模型名，也会看到绿色结果。多模态模型也缺少独立测试路径，无法验证 Vision 能力。

本设计将该页面重构为“模型注册中心”：供应商只负责凭据和端点健康，模型目录负责模型来源与能力状态，能力路由只从可用模型中选择。Ollama 不出现在该面板中，但保留后端默认和兼容逻辑。

## 目标

- 拆分“供应商连通”和“模型可用”两个概念。
- 支持内置模型目录、供应商自动发现、手动模型三种来源。
- 已配置 Key 的外部供应商默认后台刷新模型列表。
- 未知能力模型不自动进入 Vision、Long context、Reasoner 等专项槽位。
- Vision 使用真实多模态探测，不复用普通文字 ping。
- 不修改笔记 Markdown、笔记索引、IPC 安全原则或 API Key 存储方式。

## 非目标

- 不在本面板展示或管理 Ollama。
- 不自动删除用户已配置模型。
- 不根据供应商 `/models` 返回的裸 ID 自动推断复杂能力。
- 不引入新依赖。

## 页面结构

页面分为三段：供应商、模型目录、能力路由。

### 供应商

供应商区只显示外部云端供应商和自定义 OpenAI-compatible 端点，不显示 Ollama。每一行展示：

- 供应商名称。
- API Key 状态。
- Base URL。
- 健康状态。
- 上次模型刷新时间。
- 后台刷新开关。

供应商级操作包括：

- 保存或清除 Key。
- 测试供应商。
- 立即刷新模型列表。
- 添加自定义兼容端点。

“测试供应商”只验证 Key、Base URL 和认证路径可用，不声称任何模型可用。

### 模型目录

模型目录按供应商展示模型。每个模型显示：

- 模型 ID。
- 展示名。
- 来源：内置目录、供应商发现、手动添加。
- 能力标签：text、vision、tools、reasoning、long context、streaming 等。
- 验证状态：未验证、文字可用、视觉探测通过、用户确认能力、最近刷新中缺失。

未知能力模型默认只具备“文字候选”资格。若供应商只返回模型 ID，Iris 只记录“发现了这个 ID”，不自动标记 Vision、Reasoner 或 Long context。

### 能力路由

能力路由仍使用现有槽位：

- Fast
- Writer
- Reasoner
- Long context
- Vision

选择模型时按槽位过滤候选项：

- Fast / Writer 可选文字验证通过或内置文字模型。
- Vision 只显示内置 Vision 模型、视觉探测通过模型，或用户确认支持 Vision 的模型。
- Long context 只显示内置长上下文模型，或用户确认上下文能力的模型。
- Reasoner 只显示内置 reasoning/tools 模型，或用户确认对应能力的模型。

## 后台刷新

对已保存 Key 的外部供应商默认开启后台刷新。未配置供应商不刷新，Ollama 不刷新。

刷新策略：

- 应用启动后等待短暂空闲再刷新，避免阻塞启动体验。
- 每个已配置供应商定期刷新，默认周期建议为 24 小时。
- 设置页提供“立即刷新”按钮。
- 刷新结果写入本地模型目录缓存。
- 若模型后续不再出现在供应商列表中，标记为 stale，不自动删除。
- 若 stale 模型正在被能力路由使用，显示警告，但不擅自改路由。

失败处理：

- Key 错误、网络失败、供应商接口不可用只更新该供应商状态。
- 不弹全局错误。
- 不记录 API Key、请求正文中的用户内容或敏感响应。

## 数据模型

现有 `settings.llm_routing` 继续作为最终路由配置。新增或扩展一个独立模型目录缓存，用于供应商发现和用户确认信息。

模型目录条目包含：

- provider_id
- model_id
- display_name
- source：built_in、provider_discovered、manual
- first_seen_at
- last_seen_at
- last_refreshed_at
- stale
- text_verified_at
- vision_verified_at
- user_confirmed_capabilities

内置目录仍来自 `src-tauri/src/llm/model_catalog.rs`，是能力标签的可信来源。旧配置里的 `enabledModels` 兼容保留，读取时可作为手动模型纳入目录视图。

## 验证设计

### 供应商测试

供应商测试验证 Key、Base URL 和认证路径，不绑定单个模型。成功文案应明确为“供应商可连接”。

### 模型存在性验证

如果供应商 `/models` 返回可靠列表：

- 目标模型 ID 在列表中，存在性验证成功。
- 目标模型 ID 不在列表中，验证失败并提示“供应商模型列表中未找到该模型”。

如果供应商不支持可靠列表：

- fallback 到最小 chat 请求。
- 请求体必须使用用户指定的模型 ID。
- 只有该模型成功响应，才判定文字验证通过。

### 专项能力验证

Vision 使用一张内置极小测试图片，发送真实多模态请求。通过后标记 `vision_verified_at`。

Reasoner、Tools、Long context 不做默认昂贵探测。能力优先来自内置目录；未知模型需要用户手动确认后才能进入对应槽位。后续可单独扩展专项 probe。

用户手动确认和探测通过要在 UI 中分开显示，例如“用户确认支持 Vision”和“视觉探测通过”。

## 安全与隐私

- API Key 继续只存系统凭据管理器。
- 后台刷新只对用户已保存 Key 的外部供应商生效。
- 不将 Key 写入日志、数据库明文字段或普通设置文件。
- HTTP 仅允许 localhost/127.0.0.1 等本地例外；外部供应商 Base URL 仍要求 HTTPS。
- 探测请求不包含用户笔记内容。

## 测试策略

### 前端契约

- “模型与供应商”页面不展示 Ollama。
- 页面包含供应商、模型目录、能力路由三段。
- 未知能力模型不能出现在 Vision 或 Long context 候选列表。
- 用户确认能力后，模型可进入对应候选列表并显示确认来源。

### 后端探测

- `/models` 成功但不含目标模型时，模型验证失败。
- `/models` 成功且包含目标模型时，模型存在性验证成功。
- `/models` 不可用时，fallback chat 使用用户指定模型 ID。
- Vision 测试生成多模态请求体，不复用普通文字 ping。

### 后台刷新

- 只刷新已配置 Key 的外部供应商。
- 不刷新 Ollama。
- 模型消失时标记 stale，不自动删除，不自动改能力路由。

### 兼容

- 旧 `enabledModels` 可作为手动模型读取。
- 旧路由仍能 resolve。
- API Key 仍只通过系统凭据管理器读写。

## 已确认决策

- 采用“内置目录 + 供应商发现 + 手动模型”的混合方案。
- 后台刷新默认对已保存 Key 的外部供应商开启。
- 未知模型默认只允许文字能力；专项能力需要探测或用户确认。
- Ollama 从本面板完全隐藏，仅保留后端兼容逻辑。
