# 后续 AI 施工手册

## 1. 每次任务开始前

后续 AI 必须按顺序执行：

1. 阅读仓库根 `AGENTS.md` 全文。
2. 阅读本目录 `README.md`、当前阶段文档及其引用的上位规格。
3. 检查 `git status --short`，保护用户已有修改。
4. 阅读准备修改文件的完整内容。
5. 搜索相关函数、类型、IPC 和组件的所有调用处。
6. 画清当前数据流和目标数据流，再提出本次最小改动范围。
7. 先写失败测试并确认失败原因正确。

未经用户明确允许不得创建 worktree。不得因为施工计划提到后续阶段而顺手实施后续阶段。

## 2. 阶段选择

- 从 [08-implementation-plan.md](./08-implementation-plan.md) 中选择最早未完成且依赖已满足的阶段。
- 一次任务优先完成一个可验证的纵向切片。
- 若用户只要求评审、诊断或计划，不得实施代码。
- 若发现前一阶段退出条件并未满足，先修复或报告，不得假设已完成。

## 3. 测试先行模板

### 新功能

1. 写合同测试表达目标行为。
2. 运行并确认测试因能力缺失而失败。
3. 写最小实现。
4. 运行相关测试。
5. 重构重复代码。
6. 运行阶段门禁。

### Bug

1. 用最低层级稳定复现。
2. 确认修复前失败。
3. 找到根因，不以 UI 特判掩盖后端错误。
4. 修复并验证相邻边界。

### 删除旧链路

1. 先证明新链路覆盖对应行为。
2. 搜索 Rust 注册、TS wrapper、类型、组件、测试和文档引用。
3. 一次删除完整垂直链，不留无调用兼容 facade。
4. 用 `rg` 和编译验证不存在残留。

## 4. 文件修改纪律

- 修改函数/类型/组件前阅读完整文件和所有调用处。
- 优先复用现有算法；executor 化不等于重写算法。
- 不引入新依赖，除非现有依赖和标准库确实不能满足；新增依赖先检查 AGPL-3.0 兼容许可。
- Rust 内部 API 优先 `pub(crate)`。
- TypeScript 禁止 `any`，IPC 只能通过类型安全封装。
- 不在 `components/ui/` 引入 Agent 业务逻辑。
- 不修改用户 `.md` 内容作为测试准备，除非用户明确同意；测试使用 fixture/temp vault。
- 不在日志、错误、快照或测试输出中暴露笔记正文和凭据。

## 5. 架构审查清单

每次实现必须回答：

- 是否新增了第二个状态机、权限判断或证据事实源？
- 是否重新引入 scene/intent 到执行选择？
- 是否读取了 active note、tab 或光标作为 Session 隐式上下文？
- 是否有工具、Skill、MCP 或 executor 绕过 Policy Engine？
- 是否把 evidence 内容放进 system 指令层？
- 是否给简单问题增加额外模型调用或同步扫描？
- 是否让新旧数据双写？
- 是否误建 Research executor 或研究专用状态？

任一答案为“是”时，除非上位规格明确要求，否则停止并调整设计。

## 6. 安全审查清单

- 文档 read/send/cite/apply 权限是否分别判断？
- `@` 是否只限定范围而没有提升权限？
- 写回是否校验冻结 plan hash 和最新 content hash？
- Session grant 是否绑定真实 session、能力、资源和到期时间？
- 涉密内容是否完全避开普通 DB、Web、MCP 和日志？
- Provider failover 是否只覆盖允许的瞬态错误？
- API Key 是否只在实际 dispatch 时解密并 Zeroizing？

## 7. 性能审查清单

- accepted 是否在耗时工作前发出？
- 简单问答是否只有一个主模型调用？
- 是否删除了重复 Context Assemble？
- Skill/MCP/模型探测是否使用缓存并移出关键路径？
- 可以并发的 Policy、Context 计划和 Provider route 是否避免串行等待？
- 是否真实记录 dispatch、first byte、first token，而不是只记录总耗时？

## 8. 汇报格式

每次阶段性交付应向用户说明：

- 完成了哪个阶段和用户可见结果。
- 修改的核心契约与迁移影响。
- 新增或更新的测试。
- 实际运行的验证命令及结果。
- 尚未完成的下一阶段。
- 已知风险、假设或需要用户裁决的冲突。

禁止仅依据代码阅读声称“已完成”“已修复”或“已通过”。

## 9. 停止并请求裁决的条件

- 用户要求与 `AGENTS.md` 硬约束冲突。
- 需要改变 Session/当前文档解绑原则。
- 需要恢复课题研究场景。
- 需要通用 MCP 直通、代码型 Skill 或插件平台。
- 数据迁移无法在保留消息和完成结果的前提下完成。
- 涉密送模、Provider 范围或文档权限需要扩大。
- 同一事实出现两套不可兼容的数据源。

## 10. 完成前最后检查

1. 查看 `git diff --check`。
2. 审查 `git diff`，确认没有用户无关改动。
3. 运行本阶段测试和 [09-verification-and-rollout.md](./09-verification-and-rollout.md) 要求的门禁。
4. 检查日志/fixture/snapshot 不含敏感正文。
5. 更新已实现事实对应的架构、IPC 和安全文档。
6. 只有验证输出明确成功后，才向用户声称完成。
