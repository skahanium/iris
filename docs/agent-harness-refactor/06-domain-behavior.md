# 领域行为与材料使用规格

## 1. 不是场景路由

本文件定义领域材料政策和 executor 行为，不恢复互斥 scene。一个请求可以同时包含工作分析和公文输出；Harness 应组合 `authority + exemplar`，而不是在两个工作流之间二选一。

| 任务特征                   | 默认材料需要         | 自动仓库检索       |
| -------------------------- | -------------------- | ------------------ |
| 公文结构与表达             | exemplar             | 允许检索范文样本   |
| 工作规则、流程、职责、风险 | authority            | 允许检索规范依据   |
| 工作分析后形成公文         | authority + exemplar | 两类分开使用       |
| 小说创作                   | none                 | 禁止，除非明确 `@` |
| 普通聊天                   | none                 | 不因关键词自动检索 |

## 2. 公文写作 executor

### 输入

- 用户目标、文种、受众、用途和时限。
- 用户明确提供的事实和 `@` 材料。
- 相近文种的 exemplar packets。
- 涉及工作判断时的 authority packets。

### 材料职责

- Exemplar：提取结构骨架、段落功能、语气、常用表达和格式特征。
- Authority：约束工作内容、程序、禁止事项和风险；不充当语言模板。
- Reference：补充背景，必须与用户事实区分。
- Lookup：仅供查阅，不独立形成规范性结论。

### 防止范文事实污染

Executor 在起草前生成内部 `style_blueprint`，只允许包含：

- 文种结构和段落职责。
- 抽象语气、句式和格式特征。
- 不含具体人名、机构名、日期、数字和事件结论的表达模式。

最终稿事实只能来自用户输入、明确资料或可引用证据。验证器必须检测范文专有事实是否泄漏进草稿。

### 输出

- 信息不足时先列出少量高影响缺口。
- 默认输出 Markdown 草案和材料说明。
- 用户要求写回时生成确定性 patch preview，不直接修改文件。
- 同时使用 authority/exemplar 时，引用说明必须区分“内容依据”和“写法参考”。

## 3. 工作分析 executor

### 协作方式

- 维护目标、约束、事实、假设、参与方、风险和待决问题的会话摘要。
- 可先做只读检索以提出更好的问题，但不得把未确认推断写成用户事实。
- 高影响选择、规范冲突和目标变化必须回到用户。
- 不强制形式化研究流程，不创建 ResearchState。

### 输出结构

根据问题复杂度选择最小充分结构：

```text
问题理解
已知事实 / 待确认事实
相关规范依据
可选方案
风险与影响
建议及理由
下一步
```

不得为了套模板强制输出所有标题。关键要求是区分证据、规范、推断和建议。

### 规范冲突

- 若资料明确存在生效、失效或替代关系，呈现该关系及来源。
- 无法确认效力时建立 conflict group，展示冲突，不自行决定。
- 用户选择只对当前讨论有效，除非用户明确保存为长期规则。

## 4. 小说写作行为

### 允许的上下文

- 当前 Conversation 的用户与助手文本。
- 本轮明确 `@` 的文档或文件夹。
- 显式编辑器动作传入的选区或文档快照。

### 禁止的上下文

- 当前活动文档、其他 tab、同目录文件和最近打开记录。
- 自动检索的人物卡、设定集或历史章节。
- 未被 `@` 的 corpus、authority、exemplar 和 reference。

### 延续性

用户若希望跨章节保持一致，应显式 `@` 对应章节或设定。Harness 可以在对话中提示一次如何引用，但不得以“改善体验”为由自动扩大范围。

## 5. 混合请求

示例：“结合单位制度分析这个问题，并写成一份请示。”

解析结果：

```text
effect = draft
material_needs = [authority, exemplar]
effort = tool_loop or durable
```

执行顺序：

1. 使用 authority 分析内容边界和可行方案。
2. 对冲突或高影响假设向用户澄清。
3. 使用 exemplar 构造请示的结构和表达蓝图。
4. 基于已确认内容起草。
5. 如需写回，展示变更计划。

## 6. 明确排除课题研究

- 不注册 research executor。
- 不从文本自动路由到 research workflow。
- 不显示研究问题树、研究阶段控制或“开始研究”动作。
- 不保留可继续运行的旧研究 checkpoint。
- Web、多来源证据和冲突检测作为通用能力存在，但不组成专用研究产品。

## 7. 领域评测用例

至少维护以下黄金用例：

- 同文种范文与法规同时存在时，公文风格来自范文，内容约束来自法规。
- 范文中出现具体日期和机构名，最终稿未在无依据情况下复制。
- 工作讨论命中规范依据并给出引用，不把范文作为结论依据。
- 两份规范依据冲突，回答要求用户裁决而非合并出第三种规则。
- 小说请求未 `@` 时工具轨迹中没有 vault read/search。
- 小说显式 `@` 两份设定时只读取这两份及其明确范围。
- 切换当前编辑器文档不改变上述任何结果。
