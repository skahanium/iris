# Iris 1.2.5 Markdown 性能优化方案

## 一、背景与缘由

Iris 当前的 Markdown 编辑链路采用 TipTap/ProseMirror：打开 `.md` 文件时，Markdown 经 ingest 管线转换为 TipTap 可编辑内容；保存时，再从 ProseMirror 文档树序列化回 Markdown。该路线符合当前项目技术栈约束，也支撑了 Iris 已有的富文本化编辑能力，例如 wiki link、callout、任务列表、表格、图片、AI inline stream、slash command、标题折叠和查找高亮。

短期不建议切换到 CodeMirror 6，原因有三点：

1. 当前项目规范明确锁定 TipTap/ProseMirror，直接替换会牵动架构、测试、文档和产品承诺。
2. Iris 已经围绕 TipTap 建立了较多编辑扩展和 Markdown round-trip 契约，全量迁移工程量大、回归风险高。
3. 以当前主力笔记规模看，经过工程优化后，TipTap 足以覆盖大多数普通 Markdown 笔记体验。

但也必须承认，TipTap/ProseMirror 不是天然的大文档文本编辑器。它的优势在结构化富文本编辑，而不是极限 Markdown 源码性能。CodeMirror 6 在大文档、低内存、viewport 渲染和源码忠实度上有架构优势。因此，Iris 1.2.5 的目标不是证明 TipTap 可以无限扩展，而是在不切换技术栈的前提下，把现有路线优化到足够稳定、可测、可降级。

## 二、总体方向

1.2.5 的 Markdown 性能优化方向是：

- 继续保留 TipTap/ProseMirror 作为 Markdown 编辑核心。
- 不引入 CodeMirror 6，不做编辑器内核迁移。
- 把输入体验放在最高优先级，所有全文扫描、序列化、派生视图刷新都必须避开输入热路径。
- 基于文档大小和 ProseMirror 节点数建立分级策略。
- 对大文档主动降级增强能力，而不是让所有功能同时抢占主线程。
- 用性能基线和契约测试守住优化成果，避免后续功能把成本重新加回热路径。

长期上，Iris 仍应保留“源码模式 / 大文档模式”的技术退路；但 1.2.5 不执行该迁移，只优化当前 TipTap 路线。

## 三、当前基础

项目中已经具备一些重要性能基础，1.2.5 应继续复用，而不是重复造轮子：

- `editor-ingest-async.ts` 已支持超过 50KB 的 Markdown ingest 进入 Worker。
- `editor-html-cache.ts` 已提供按 path + namespace + digest 缓存 TipTap HTML 的能力。
- `AppEditorWorkspace.tsx` 已通过 `surfaceRecords` 保留多个 ready editor surface，支持已打开 tab 快速回切。
- `TipTapEditor.tsx` 已使用 `shouldRerenderOnTransaction: false`，避免每次 transaction 都触发 React 重渲染。
- `useEditorSave.ts` 已使用 1200ms debounce，保存时才序列化 Markdown。
- `editor-pm-serialize.ts` 已提供 ProseMirror Markdown serializer 热路径，失败时才 fallback 到 HTML turndown。
- `EditorOutline.tsx` 已使用虚拟列表渲染 outline。

因此，1.2.5 的重点不是新增一套缓存系统，而是校准和强化现有热路径。

## 四、性能目标与文档分级

Iris 1.2.5 应按文档规模建立明确性能预期：

| 等级   | Markdown 大小 | 目标体验                             |
| ------ | ------------- | ------------------------------------ |
| small  | <= 50KB       | 完整增强能力，打开和输入都应流畅     |
| medium | 50KB - 150KB  | Worker ingest，完整能力基本保留      |
| large  | 150KB - 500KB | 输入优先，派生功能延迟或降频         |
| huge   | > 500KB       | 避免卡死和数据损坏，增强能力主动降级 |

除文件大小外，还应统计 ProseMirror doc 节点数。若节点数、表格复杂度、图片/媒体嵌入或 decorations 数量明显偏高，即使文件小于阈值，也应提升到更高性能等级。

建议目标：

- `<=150KB`：打开、编辑、保存无明显卡顿。
- `150KB-300KB`：输入保持跟手，派生信息允许轻微延迟。
- `300KB-500KB`：打开可显示 loading surface，输入优先，实时增强功能可降级。
- `>500KB`：不承诺完整 WYSIWYG 增强体验，但必须避免长时间无响应。

## 五、核心优化策略

### 5.1 打开链路

目标：减少从点击笔记到首帧可编辑之间的主线程压力。

- 保留现有 warm prepare、prepared editor HTML 和 editor surface 缓存。
- 大文档 ingest 必须继续走 Worker，禁止回退到主线程同步 ingest。
- 将 Markdown ingest Worker 从“每次创建后立即 terminate”优化为受控复用通道，降低连续打开大文档时的 Worker 创建成本。
- Worker 请求必须带 requestId，过期结果必须丢弃，避免快速切换 tab 时旧文档覆盖新文档。
- `editor-html-cache` 命中路径应尽量 O(1) 返回，不应每次读取都 DOMParser 全量扫描 HTML。
- 坏 HTML 防护应保留在写入缓存和 cache format version 变更时，而不是每次命中都重复执行。

### 5.2 输入热路径

目标：用户每次输入时只做必要工作。

- `onUpdate` 只允许做 dirty 标记和轻量调度。
- 禁止在输入同步路径中调用 `editor.getHTML()`、`editor.getText()`、Markdown serializer 或全量 outline 构建。
- body stats 当前会扫描 `doc.textContent`，应按文档等级调整：
  - small / medium：保持短 debounce。
  - large：改为 idle callback 或更长 debounce。
  - huge：只在打开、保存后或用户明显停止输入后刷新。
- HTML cache 写入继续 debounce；large / huge 文档延长写入延迟，并只在空闲时执行 `editor.getHTML()`。
- 插件 decorations 应限制数量和刷新频率，AI source highlight、find highlight、wiki media preview 等不应在大文档中无限增长。

### 5.3 派生视图

目标：outline、查找、统计、AI 上下文等派生能力不能和正文编辑抢帧。

- `EditorOutline` 已经虚拟化，但 `outlineFromDoc` 仍会扫描 doc；large 文档改为空闲刷新，huge 文档默认只在打开后生成一次，并提供手动刷新。
- `EditorFindReplaceBar` 当前 query 变化会同步扫描 doc；large / huge 文档改为 debounce 搜索。
- 查找高亮应设置最大 decorations 数量；超过上限时仍显示匹配总数或提示截断，但不渲染所有高亮。
- hidden editor surface 不应触发 outline、stats、find highlight、media eager render 等派生工作。
- AI 面板调用 `getNoteContent()` 会触发 live Markdown 序列化；应只在真正发送请求、生成 artifact 或执行 patch 前调用，不在普通 render 或 hover 阶段调用。

### 5.4 保存与序列化

目标：保存可靠优先，同时避免保存期间引发连锁重计算。

- 保留 `useEditorSave` 的 1200ms debounce。
- 保存时继续优先使用 `editorDocToMarkdown` 的 ProseMirror serializer。
- 若 serializer fallback 到 `editorBodyHtmlToMarkdown`，需要记录非敏感 telemetry 或开发日志计数，用于发现慢路径。
- large / huge 文档保存期间按优先级调度：
  1. 写入 `.md` 文件。
  2. 更新 last saved snapshot。
  3. 版本快照。
  4. HTML cache 写入。
  5. outline、stats 等派生刷新。
- 保存成功后复用已序列化 Markdown，避免版本快照再次序列化全文。

### 5.5 Editor Surface 缓存

目标：保留已打开 tab 的快速回切，同时控制内存。

- 继续使用 `AppEditorWorkspace` 的 path-stable `surfaceRecords`。
- 不新增 `EditorViewCache`、WeakRef editor pool 或第二套 editor 生命周期 owner。
- `READY_SURFACE_RETAIN_LIMIT` 当前为 8，应通过实际内存和回切数据确认是否合理。
- hidden surface 应保持 DOM/Editor 状态，但禁止执行非必要派生更新。
- 对 large / huge 文档，必要时降低保留数量，优先保留最近使用和当前 pending open 文档。

### 5.6 媒体和复杂节点

目标：大文档中复杂节点不应拖累正文输入。

- 图片、wiki media embed、复杂 NodeView 默认延迟加载。
- large / huge 文档中禁用非可见媒体的 eager render。
- 表格应限制实时增强能力；大型表格优先保证基础编辑，不强求复杂 hover toolbar 或全表扫描。
- AI stream 节点完成后应及时收敛为普通内容，避免长期保留高成本临时节点。

## 六、测试与验收

### 6.1 契约测试

1.2.5 应补齐以下测试：

- 大文档 ingest 必须走 Worker。
- cache hit 不执行 DOMParser 全文扫描。
- `onUpdate` 不同步触发 `getHTML()`、`getText()` 或 Markdown serializer。
- large / huge 文档下 outline、stats、find 必须 debounce、idle 或有数量上限。
- hidden editor surface 不触发派生刷新。
- PM serializer fallback 必须可观测，不能无声退回慢路径。
- 保存后的 Markdown round-trip 行为保持不变。

### 6.2 性能基线

使用合成 fixture，不使用真实用户笔记正文：

- 50KB 纯段落文档。
- 150KB 长列表文档。
- 300KB 多标题、多链接、多 callout 文档。
- 500KB 混合表格、代码块、图片链接文档。

每组记录：

- 打开到 loading surface 的时间。
- 打开到 first editor frame 的时间。
- 连续输入 30 秒的 long task 数量。
- 保存序列化耗时。
- tab 回切耗时。
- 内存峰值。
- 是否触发 serializer fallback。

性能 trace、截图和日志不得包含用户笔记正文、frontmatter、prompt、API Key、Token、真实路径或解密后的涉密内容。

## 七、实施顺序

### Phase 0：基线与防线

- 固定性能 fixture。
- 扩展 `docs/ops/performance-guide.md` 的 Markdown 编辑器专项采样流程。
- 增加 source contract，禁止在输入热路径加入全文工作。

### Phase 1：低风险热路径治理

- 调整 body stats 调度。
- 调整 HTML cache 写入和命中策略。
- 为 outline / find 增加文档等级感知。
- 确保 hidden surface 不做派生刷新。

### Phase 2：Worker 与缓存强化

- 复用 Markdown ingest Worker。
- 明确 stale request 丢弃策略。
- 扩展 prepared editor HTML 的命中记录和预算检查。
- 记录 PM serializer fallback 次数。

### Phase 3：大文档降级

- 引入文档性能等级。
- large / huge 文档默认降低 outline、find highlight、media eager render 和复杂 decorations。
- 在 UI 上保持克制，不弹打扰式提示；只在必要位置显示状态，例如“目录已延迟刷新”。

## 八、非目标

1.2.5 不做以下事情：

- 不引入 CodeMirror 6。
- 不更换 Markdown 解析器。
- 不把 `.md` 内容写入专有格式。
- 不新增全局状态库。
- 不创建第二套 editor cache / editor pool。
- 不把 database、多维表格纳入本轮正文编辑器优化。
- 不为了性能牺牲 Markdown round-trip 正确性。

## 九、风险与取舍

- TipTap/ProseMirror 的结构化文档模型决定了它不适合无限大的 Markdown 文档，优化只能把常规笔记体验做到稳定，不能消除与 CodeMirror 6 的架构差距。
- 大文档降级会让部分增强功能延迟或减少，这比输入卡顿和应用无响应更可接受。
- HTML cache 命中减少重复扫描后，需要依赖 cache format version 和写入时校验来保证安全，后续修改 ingest 语义时必须 bump cache version。
- Worker 复用会带来生命周期复杂度，必须通过 requestId、terminate 时机和测试控制风险。

## 十、结论

Iris 1.2.5 应继续优化当前 TipTap/Markdown 技术路线，而不是切换编辑器内核。合理目标是让 50KB-300KB 的主力笔记体验稳定流畅，让 300KB-500KB 的大笔记可用，并为 500KB 以上文档建立明确降级策略。

这条路线能够最大化复用现有工程资产，同时避免过早承担 CodeMirror 迁移成本。后续若 Iris 的真实用户文档持续逼近 MB 级，或 Markdown 源码忠实编辑成为核心竞争点，再重新评估 CodeMirror 6 源码模式或大文档模式。
